# File-System Implementation

파일 시스템 구현



# Allocation Methods

할당 방법

* **연속 할당**
* **연결 할당**
* **색인 할당**



## 연속 할당 (1)

각 파일들을 디스크의 **연속적인 공간을 차지하도록 요구**

* 연속된 블록의 이동 시에 **헤드의 이동이 필요하지 않다**
* 파일에 **접근하기 쉽다**

<img src="../capture/스크린샷 2019-06-13 오후 11.02.27.png" width=500>

> 문제점 : 파일 크기가 **정해져 있지 않다**



## 연속 할당 (2)

순차적 접근과 직접 접근 두 가지 모두 지원한다

* **동적 공간 할당 문제 (dynamic storage-allocation problem)**
  * 새로운 파일을 위한 가용 공간을 찾는 일
  * 단편화 : 파일들이 할당되고 삭제되면서 빈 디스크 조각들이 생긴다
  * 파일이 필요한 공간 확장이 어렵다
  * 공간 낭비



## 연결 할당 (1)

각 파일들이 디스크 블록들의 **리스트에 연결되며** 디스크 블록들은 디스크 내의 어느 곳에 흩어져 있어도 된다

<img src="../capture/스크린샷 2019-06-13 오후 11.09.57.png" width=500>



## 연결 할당 (2)

**문제점**

* 순차적 접근 파일에만 효과적이다 => **직접 접근 기능을 지원할 수 없다**
* **포인터들을 위한 공간이 필요**
* 실뢰성 문제 발생 가능



## 연결 할당 (3)

연결 할당의 변형 : **파일 할당 테이블(FAT)의** 사용

* MS-DOS
* 각 디스크 블록 내의 한 개 항목을 가지고 있고 **블록 번호에 의해 참조**

<img src="../capture/스크린샷 2019-06-13 오후 11.15.17.png" width=500>



## 색인 할당 (1)

모든 포인터들을 하나의 장소인 **색인 블록으로** 관리함으로써 **연결 할당의 문제 해결**

* 각 파일에 대해 색인 블록 관리

<img src="../capture/스크린샷 2019-06-13 오후 11.18.24.png" width=500>



## 색인 할당 (2)

* **직접 접근이** 가능하다
* 공간의 **낭비가 심하다** : 색인 블록의 포인터 부담
* **커다란 파일을** 표현할 수 없다
  * 해결 방법들
    * <u>연결 기법 (linked scheme)</u>
    * <u>다중 단계 색인 (multilevel index)</u>
    * <u>결합 기법 (combined scheme)</u>



## 색인 할당 (3)

**연결 기법 (linked scheme)**

* 큰 파일을 수용하기 위해서는 **여러 개의 색인 블록들을 직접 연결**
* 파일 크기의 **제한이 없다**



## 색인 할당 (4)

**다중 단계 색인 (multilevel index)**

* 분리된 색인 블록들을 사용하여 **다른 색인 블록에 연결**
* Two-level index

<img src="../capture/스크린샷 2019-06-13 오후 11.23.17.png" width=500>



## 색인 할당 (5)

**결합 기법 (combined scheme)** : UNIX에서 사용되는 기법

<img src="../capture/스크린샷 2019-06-13 오후 11.32.15.png" width=500>

> Direct blocks : 데이터를 바로 찾아간다
>
> N indirect : n이 증가할수록 파일 크기가 증가하게 되고, 접근 속도는 감소한다. 왜냐하면 색인을 거쳐서 접근하기 때문이다.
