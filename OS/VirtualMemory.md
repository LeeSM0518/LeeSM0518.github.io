# Page Replacement

페이지 교체



<img src="../capture/스크린샷 2019-06-12 오후 9.58.14.png" width=500>

다중 프로그래밍의 정도가 높아짐에 따라 **과할당 발생**

* 자유 프레임 리스트에 비어 있는 프레임이 **없는 상황이 발생**



**해결책**

* 사용자 프로그램의 중단
* 프로세스의 swap out

=> <u>프로세스 교체 또는 페이지 교체(page replacement) 요구</u>



## 기본적인 페이지 교체 (1)

<img src="../capture/스크린샷 2019-06-12 오후 10.00.12.png" width=500>

1. 희생자 페이지를 내보냄 : 현재 **사용되고 있지 않는** 프레임

2. 비트 변경 : 페이지 테이블을 수정하여 프레임을 **빈 상태로 만든다**
3. 원하는 페이지를 읽어 들임
4. 새로운 페이지를 위한 페이지 테이블 재설정

> 빈 프레임이 없는 경우에 디스크를 두 번 접근해야 하므로 페이지 부재 처리 시간(page fault service time)이 **2배** 소요된다

## 기본 페이지 교체 (4)

**가장 낮은 페이지 폴트 비율을** 얻을 수 있는 방향으로 페이지 교체 알고리즘 선정

* **페이지 교체 알고리즘들**

  * 계수-기반 페이지 교체 알고리즘



## FIFO 페이지 교체 (1)

**FIFO (First-In-First-Out) Page Replacement Algorithm**

* 가장 간단한 페이지 대체 알고리즘
  * 페이지가 교체되어야 할 때 **가장 오래된** 페이지를 선택

<img src="../capture/스크린샷 2019-06-14 오후 1.50.32.png">



## FIFO 페이지 교체 (2)

**Belady의 모순 (Belady's anomaly)**

* 프레임의 수를 증가시켰는데도 오히려 페이지 부재율이 더 증가하는 현상 발생

<img src="../capture/스크린샷 2019-06-14 오후 1.53.21.png" width=500>



## 최적 페이지 교체

**Optimal Page Replacement Algorithm**

* 앞으로 가장 오랜 기간 동안 **사용되지 않을** 페이지를 교체
  * 페이지 폴트 비율이 가장 낮다
  * Belady의 모순이 발생하지 않는다

<img src="../capture/스크린샷 2019-06-14 오후 1.55.41.png">

> 현실적으로 최적 페이지 교체 알고리즘은 구현하기 어렵다
>
> : 참조열에 대한 **미래의 지식을** 알기 어렵기 때문이다. 주로 비교 연구를 위해 사용



## LRU 페이지 교체 (1)

**Least Recently Used (LRU) Page Replacement Algorithm**

* 최적 알고리즘의 근사치
  * 가까운 미래의 근사치로 **가장 최근의 과거를** 사용
  * 오랜 기간 동안 **사용되지 않은** 페이지를 교체

<img src="../capture/스크린샷 2019-06-14 오후 1.58.29.png">



## LRU 페이지 교체 (2)

* 자주 사용되며 인정받는 알고리즘
* Belady's anomaly가 없다
* LRU 페이지 교체 알고리즘의 구현
  * 카운터(counter) 사용
  * 스택(stack) 사용
  * **하드웨어의 지원 필요** : 클럭 또는 스택의 수정이 기억 장소 참조 시 



## LRU 페이지 교체 (3)

**카운터(counter)의 사용**

* 각 페이지 항목 마다 **사용 시간 필드를** 넣고 CPU에 **논리 클럭이나 카운터를** 추가
  * 메모리 접근 시마다 시간 증가
  * 페이지 참조 시에 **시간 레지스터의 내용을** 페이지의 사용 시간 필드에 복사
  * **가장 작은 시간 값을** 가진 페이지 교체

<img src="../capture/스크린샷 2019-06-14 오후 2.08.10.png">



## LRU 페이지 교체 (4)

**스택 사용**

* 페이지 번호의 스택을 유지
  * 페이지가 **참조할 때마다** 페이지 번호를 스택의 **top에** 유지
  * **bottom에** 있는 페이지가 **가장 오래 전에 사용된** 페이지

<img src="../capture/스크린샷 2019-06-14 오후 2.10.37.png" width=500>



## LRU 근사 페이지 교체 (1)

* LRU 페이지 교체를 위한 **하드웨어 지원 부족**

  * 많은 시스템들이 **참조 비트(reference bit)의 형태 지원하게 됨**

* **부가적 참조 비트 알고리즘**

  * 각 페이지에 대해 **8비트 유지**
  * 타이머 인터럽트가 제어를 운영 체제에 넘기면 **참조 비트를 8비트의 높은 자리로 이동하고** 다른 비트들을 하나씩 **오른쪽으로 이동**
  * 가장 낮은 번호를 가진 페이지를 대치

  <img src="../capture/스크린샷 2019-06-14 오후 2.15.14.png" width=200>



## LRU 근사 페이지 교체 (2)

**2차 기회 알고리즘** : 선입선출 알고리즘의 변형

<img src="../capture/스크린샷 2019-06-14 오후 2.17.58.png" width=500>

1. 포인터는 참조 비트가 0인 페이지를 발견할 때까지 포인터를 전진시킨다
2. 포인터가 전진하면서 참조 비트를 **0으로 바꾼다**
3. 0인 페이지를 발견하면, 그 페이지는 교체되고 새로운 페이지가 해당 위치에 삽입된다



## 페이지 버퍼링 알고리즘

특정 페이지 교체 알고리즘에 **다른 처리 과정이** 추가되어 사용된다

* **자유 프레임 풀(pool)의 유지**
  * 페이지 폴드가 일어날 때 희생될 프레임으로 선택된 페이지는 하나의 **자유 프레임으로 들어간다**
  * 희생된 페이지가 나가기를 가디리지 않고 프로그램을 **빠르게 시작할 수 있다**
  * 자유 프레임의 저장소를 유지하면서 각 프레임에 **어느 페이지가 있는가를 기억**
* **변경된 페이지들의 리스트를 유지**
  * 변경된 페이지를 선택하여 디스크에 기록하고 변경 비트를 0으로 설정
  * 페이지가 변하지 않았을 가능설을 높인다



# Allocation of Frames

프레임 할당

* 방법
  * 사용 가능한 모든 프레임을 **자유 프레임 리스트 (free frame list)로** 유지
  * 페이지 폴트가 발생할 때마다 프레임을 할당



## 최소 프레임 수

기억 장소 할당에는 제한이 있어서 **총 유효 프레임 수보다** 할당할 수 없지만, 할당 가능한 최소의 프레임이 있다.

* **최소한의 프레임을 할당해야 하는 이유**
  * 프로세스에 할당되는 프레임 수가 줄어들면 페이지 폴트율이 증가하고 프로세스 실행이 늦어진다
  * 명령어 집합 구조(instruction set architecture)



## 할당 알고리즘 (1)

**Fixed allocation**

* 균등 할당 (equal allocation) : 모든 프로세스에 **똑같은 프레임을 할당하고** 남은 프레임은 **가용 프레임 버퍼 저장소로** 사용
* 비례 할당 (proportional allocation) : 프로세스의 **크기에 비례하여** 각 프로세스에 프레임을 할당



**예) 62개 프레임, 10 페이지 프로세스, 127 페이지 프로세스**

* **균등 할당** : 각 프로세스에 31개 프레임
* **비례 할당**
  * 10 페이지 프로세스 : 10 / 137 * 62 = 4 프레임
  * 127 페이지 프로세스 : 127 / 137 * 62 = 57 프레임



## 할당 알고리즘 (2)

**Priority allocation**

* 우선 순위의 고려 : 우선 순위가 높은 프로세스에 **많은 기억 장소를 할당하여** 수행 속도를 높이는 것이 바람직
* 비례 할당법 사용하되 할당하는 프레임 수를 프로세스의 상대적 크기가 아닌 우선 순위를 고려하거나 둘 모두를 고려하도록 한다
* 우선 순위가 높은 프로세스가 교체할 프레임을 우선 순위가 낮은 프로세스에 할당된 프레임 중에서 선택



## 전역 대 지역 할당

**전역 교체 (global replacement)** : 프로세스가 교환할 프레임을 그 프레임이 현재 다른 프로세스에게 <u>할당되어 있어도 그 에 상관없이</u> 전체 프레임 중에서 하나를 선택해서 그 프레임을 사용할 수 있도록



**지역 교체 (local replacement)** : 각 프로세스가 그 프로세스에 <u>할당된 프레임들 중에서만</u> 선택한다. 프로세스에 할당된 프레임의 <u>수는 변하지 않는다.</u>



# Thrashing

스래싱



## 스래싱의 원인 (1)

운영체제는 **CPU의 이용률을 감시한다**

* CPU 이용률이 너무 낮아지면 운영체제는 **다중 프로그래밍의 정도를 증가시키기 위해** 새로운 프로세스를 시스템에 추가한다
  * 프로세스 증가 = CPU 이용률
* **스래싱 (Thrashing)**
  * 프로세스가 실제로 사용하는 프레임 수만큼의 프레임을 가지지 못해 계속적으로 페이지 폴트가 발생함으로 인해 **계속적으로 페이지 교체가 일어나는 현상**
  * 실행 시간 < 페이지 교체 시간



## 스래싱의 원인 (2)

다중 프로그래밍이 증가하면 CPU 이용률이 증가하다가 스래싱이 일어나면 이용률이 급격히 저하한다

* **프로세스 실행보다 페이지 교체에 보내는 시간이 더 크기 때문**

<img src="../capture/스크린샷 2019-06-14 오후 3.18.26.png" width=500>

**스래싱 현상의 방지**

* 각 프로세스가 <u>필요로 하는 프레임을 제공해야 한다</u>
* 프로세스 실행의 <u>지역성 모델 (locality model)을</u> 기반으로 한다

## 페이지 폴트 빈도

**페이지 폴트 비율 조절**

* 페이지 폴트 비율이 높다 = 더 많은 프레임을 필요
* 페이지 폴트 비율니 낮다 = 너무 많은 프레임을 할당
* 페이지 폴트 비율의 **상한과 하한을** 두고 상한을 넘으면 프레임을 **더 할당하고** 하한보다 낮으면 **프레임을 회수**
* 페이지 폴트 비율이 증가되는데 유효 프레임이 없으면 한 프로세스를 선택하여 중지

# Allocating Kernel Memory

커널 메모리 할당



커널 메모리는 사용자 모드 프로세스에게 할당해 주기 위한 페이지 리스트와는 별도의 **자유 메모리 풀에서 할당받는다**

* 커널은 **다양한 크기의 자료 구조를 위해 메모리를 할당 받는다**
  * 자료 구조들은 **페이지 크기보다 작은 크기를 갖기도 한다**
* 물리 메모리에 직접 접근하는 특정 하드웨어 장치는 물리적으로 **연속적인 메모리를 필요로 하기도 한다**

## 버디 시스템 (1)

물리적으로 **연속된 페이지들로 이루어진 고정된 크기의 세그먼트로부터** 메모리를 할당 받는다

* 메모리는 2의 거듭제곱 할당기에 의해 **2의 거듭제곱 단위로 할당된다**

## 슬랩 할당

* **슬랩 (Slab)**
  * 하나 또는 그 이상의 물리적으로 연속된 페이지들로 구성
* **캐시 (Cache)**
  * 하나 혹은 그 이상의 **슬랩들로 구성**
  * 각 캐시는 **단일 타입의 객체를** 포함한다
* **커널 객체를 저장하기 위해 캐시를 사용한다**
