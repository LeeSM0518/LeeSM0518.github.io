# 02. Critical Section Problem (임계 구역 문제)

**임계 구역 (critical section)**

* 다른 프로세스와 **공유하는 변수를 변경하거나,** 테이블을 갱신하거나 파일을 쓰거나 하는 등의 작업을 수행한다.
* 임계 구역의 실행은 시간적으로 **상호 배타적**
  * 한 프로세스가 임계 구역에서 수행하는 동안에는 다른 프로세스들은 임계 구역에서 실행되지 않도록 한다.



**임계 구역 문제는 프로세스들이 협력할 때 사용할 수 있는 프로토콜을 설계 하는 것이다.**

* **진입 구역 (entry section)**
  * 자신의 임계 구역으로 진입하려면 **진입 허가를 요청해야 한다.**
* **나머지 구역 (remainder section)**
  * 코드의 나머지 부분

<img src="../capture/스크린샷 2019-04-23 오후 11.53.38.png" width="500">



## 임계 구역 문제 해결안

* **상호 배제 (mutual exclusion)**

  : 한 프로세스가 임계 구역에서 실행된다면 다른 프로세스들은 임계 구역에서 실행될 수 없다.

* **진행 (progress)**

  : 자기의 임계 구역에서 실행되는 프로세스가 없고, 그들 자신의 임계 구역으로 진입하려고 하는 프로세스들이 있다면, 나머지 구역에서 실행 중이지 않은 프로세스들만 다음에 임계 구역에 진입할 프로세스를 결정하는 데 참여할 수 있으며, 이 선택은 무한하게 연기될 수 없다.

* **한계 대기 (bounded waiting)**

  * 프로세스가 자기의 임계 구역에 진입하려는 요청을 한 후부터 그 요청이 허용될 때 까지 다른 프로세스들이 임계 구역에 진입하도록 허용하는 횟수에 한계가 있어야 한다.



## 운영체제 내에서의 임계 구역

* 운영체제 내에서 임계 구역을 다루기 위해 **선점형 커널과 비선점형 커널의** 두 가지 일반적인 접근법이 사용된다.
  * **비선점형 커널**
    * 커널 모드에서 수행되는 프로세스의 **선점을 허용하지 않는다.**
    * 자발적으로 CPU의 제어를 양보할 때까지 계속 수행된다.
    * 경쟁 조건을 염려할 필요가 없다.
  * **선점형 커널**
    * 프로세스가 커널 모드에서 수행되는 동안 **선점되는 것을 허용한다.**
    * 실시간 프로그래밍에 적당하다.
    * 경쟁 조건을 염려하고 설계해야 한다.



# 03. Peterson's Solution (피터슨의 해결안)

각 프로세스에 P0, P1 으로 번호 부여한다.



## 알고리즘 1

**공동 변수 turn 사용.**

<img src="../capture/스크린샷 2019-04-24 오전 12.17.07.png" width="500">

* 상호 배제 조건을 충족한다.

* **진행 요건을 충족하지 못한다.**

  : turn이 0이고 P0가 나머지 구역에 있으면 P1이 임계 구역에 들어갈 준비가 되어 있어도 임계 구역으로 들어갈 수 없다. **즉, P1은 P0가 실행되기 전에 실행되지 않는 문제점이 있다.**

* 프로세스 상태에 관한 정보를 충분히 유지하지 못하고 있다.



## 알고리즘 2

**배열 flag 사용**

<img src="../capture/스크린샷 2019-04-24 오전 12.21.39.png" width="500">

* **flag[2]** : 임계 구역으로의 진입 여부를 확인하는 상태. 프로세스가 각자 하나씩 갖고 있는다.
* 상호 배제 조건을 충족한다.
* **진행 조건을 충족되지 않는다.**
  * 두 프로세스가 동시에 시작되면 flag 값이 둘다 true 이기 때문에 진입 구역에서 무한 반복하게 된다.

  

## Peterson's solution (피터슨의 해결안)

**알고리즘 1과 2의 개념을 결합**

<img src="../capture/스크린샷 2019-04-24 오전 12.30.36.png" width="350">

* **3가지 요건 모두 충족**
  * 상호 배제
    * 두 프로세스가 동시에 실행된다 해도 flag는 둘다 true 이지만 turn 이 1이거나 0 이기 때문에 한 프로세스는 임계 구역으로 들어갈 수 있게 된다.
  * 진행 조건
    * P0가 요청했을 때 이미 P1이 임계 구역을 수행 중이면, flag[1] = true 이고 turn = 1 이므로 P0는 기다리다가, P1가 임계 구역을 벗어나는 순간 flag[1] = false 가 되어 P0 가 임계 구역에 진입하게 된다.
  * 한계 대기
    * P0는 P1이 기껏해야 한번 임계 구역에 들어간 후에는, P1이 나머지 구역을 수행 중이라도 임계 구역에 들어갈 수 있다. **즉, 대기 시간이 한없이 길어지지 않는다.**



# 04. Synchronization Hardware (동기화 하드웨어)

하드웨어에서부터 소프트웨어 기반 API를 망라한 기법을 사용한 임계 구역 문제 해결책들

* **동기화 하드웨어**
* **뮤텍스 / 세마포**
* **모니터**

> 임계 구역을 보호가기 위해서 **락(lock)** 을 사용하는 것에 기반을 둔다.

<img src="../capture/스크린샷 2019-04-24 오후 1.07.54.png" width="300">



* 임계 구역 문제는 단일 처리기 환경에서 공유 변수가 변경되는 동안 **인터럽트 발생을 허용하지 않음으로써** 간단히 해결할 수 있다.
  * **atomic hardware instructions (원자같은 하드웨어 명령어)**
    * 하드웨어 명령어의 중요한 특징 중 하나가 **인터럽트 되지 않고 하나의 단위로 처리**
    * 임계 구역 문제를 보다 쉽고 **효율적으로 해결**



## test_and_set 명령어

```java
boolean test_and_set (boolean *target)
{
  boolean rv = *target;
  *target = true;
  return rv;
}

do {
  while (test_and_set(&lock));	// do nothing
  
  // critical section
  
  lock = false;
  
  // remainder section
  
} while (true)
```



## compare_and_swap 명령어

```java
int compare_and_swap (int *value, int expected, int new_value)
{
  int temp = *value;
  if (*value == expected)
    *value = new_value;
  return temp;
}

do {
  while (compare_and_swap(&lock, 0, 1) != 0);	// do nothing
  
  // critical section
  
  lock = 0;
  
  // remainder section
  
} while (true)
```



# 05. Mutex Locks (뮤텍스 락)

* 임계 구역을 보호하고 경쟁 조건을 방지하기 위해 **mutex 락을** 사용한다.
* Mutex 락은 boolean 변수 **available** 을 가지며, 이 변수 값이 **락의 가용(사용) 여부를** 표시한다.

<img src="../capture/스크린샷 2019-04-24 오후 2.36.47.png">



* **바쁜 대기 (busy waiting)에 의한 구현 => spinlock**
  * 한 프로세스가 임계 구역에 있는 동안 **임계 구역에 진입하려는 다른 프로세스들은** acquire() 함수를 호출하는 반복문을 **계속 실행해야 한다.** (Mutex 락이 가용해지기를 기다리면서)
  * 다중 프로그래밍 시스템에서 **CPU 낭비**
  * 스핀락의 장점은 **문맥 교환을** 전혀 필요로 하지 않는다. (즉시 처리 가능)
  * **다중 처리기 시스템(즉각 서비스를 위해)** 에서 유용하게 이용된다.



# 06. Semaphores (세마포)

mutex와 유사하게 동작하지만 프로세스들이 자신들의 행동을 더 정교하 게 동기화할 수 있는 방법



## 세마포

**세마포(semaphore) = 정수 변수 (OS만이 건드릴 수 있는 변수)**

* 두 개의 표준 원자적 연산으로만 접근 가능

  **wait( ) & signal( ), originally called P( ) and V( )**

  ```java
  wait(S) {
    while (S <= 0);		// busy wait
    S--;
  }
  
  signal(S) {
    S++;
  }
  ```

* 세마포의 정수 값을 수정하는 것은 **원자적으로 수행한다. (wait, signal 은 시스템 호출이기 때문에 세마포는 OS만이 건드릴 수 있다.)**



## 세마포 사용법

* **카운팅 세마포와 이진 세마포**

  * **카운팅 세마포의** 값은 제한이 없다.
    * 유한한 개수를 가진 자원에 대한 접근을 제어
  * **이진 세마포의** 값은 0과 1 사이의 값만 가능하다.
    * Mutex 락과 유사하게 동작한다.

* **동기화 문제를 해결하기 위해 세마포를 사용할 수 있다.**

  * S2 는 S1 이 끝난 뒤에만 수행되어야 한다.

    <img src="../capture/스크린샷 2019-04-24 오후 2.49.51.png">



## 세마포 구현 (1)

* 한 프로세스가 임계 구역에 있으면 이 임계 구역에 진입하려는 프로세스는 진입 코드에서 계속 반복해야 한다. => **다중 프로그래밍 시스템에서 CPU 낭비**
* **다중 프로세서 시스템 에서는 유용하게** 이용될 수 있다.



## 세마포 구현 (2)

**바쁜 대기 (busy waiting)에 의한 구현 => spinlock**

* wait() & signal() 연산의 변경

  * block & wakeup 연산 이용
    * **block** : 프로세스 지연, block 연산에 의해 프로세스 **자신을 봉쇄(block)** 시킴
    * **wakeup** : 중지된 다른 프로세스를 재시작, 다른 프로세스의 signal 연산내의 wakeup 연산으로 재시작.

## 우선순위 역전

**높은 우선순위 프로세스가 낮은 우선순위 프로세스에** 의해 접근되고 있는 커널 데이터를 읽거나 변경할 필요가 있을 때 스케줄이 문제가 생긴다.

* 예 : 우선순위가 L < M < H 인 3개의 프로세스
  * H 는 L이 사용중인 자원 R을 필요로 하고 M이 실행가능 상태가 되어 L을 선점
* **우선순위 역전 (priority inversion) 문제**
  * 두 개의 우선순위만 제공하는 방법
  * **우선순위 상속 프로토콜 (priority-inheritance protocol)을 구현**
    * 자기보다 높은 우선 순위의 프로세스가 자신의 자원에 접근하면 문제가 된 자원의 사용이 끝날 때까지 **더 높은 우선순위를 상속받는다.**
    * 자원 사용이 끝나면 **원래 우선순위로 되돌아간다.**

# 08. Monitors (모니터)

* 세마포가 프로세스들 간의 **동기화를 위해서 편리하고 효과적으로** 쓰일 수 있지만, 잘못 사용하면 발견하기 어려운 **타이밍 오류를 발생시킨다.**
* 고급 언어 구조물의 개발

  * 한 순간에 하나의 프로세스만 모니터 안에서 활동하도록 보장

# 09. Deadlocks (교착상태)

* **교착 상태란?**
  * 대기 중인 프로세스들이 결코 다시는 그 상태를 변경시킬 수 없으면 이런 상황을 교착 상태라고 부른다.



## 시스템 모델

* 시스템은 경쟁하는 프로세스들 사이에 분산되어 있는 유한한 수의 자원들로 구성된다.
  * CPU 주기, 메모리 공간, 파일, 입출력 장치 등
* **프로세스 자원 이동 순서**
  * **요청** : 자원을 얻을 때까지 대기한다.
  * **사용** : 자원에 대한 작업을 수행한다.
  * **방출** : 자원을 방출한다.



# 10. Deadlock Characterization (교착상태  특징)

## 교착상태의 특징

교착 상태에 있는 **프로세스들은 결코 실행을 끝낼 수가 없으며** 시스템 자원들이 묶여 있어서 **다른 작업들을 시작하는 것도 불가능하다.**

* 4가지 조건이 동시에 만족할 때 교착 상태가 된다.
  * **상호 배제** (Mutual exclusion) : 한번에 한 프로세스만 자원 사용
  * **점유하며 대기** (Hold and wait) : 프로세스는 최소한 하나의 자원을 점유하고, 현재 다른 프로세스에 의하여 점유된 자원을 추가로 얻기 위해 대기하고 있어야 함
  * **비선점** (No preemption) : 자원들을 선점하지 못한다. (뺏을 수가 없다.)
  * **순환 대기** (Circular wait) : P0 는 P1 이 점유한 자원을 대기하고 P1 은 P2 가 점유한 자원을 대기하고 P2 는 P0 가 점유한 자원을 대기한다.

* **그래프에 사이클이 없다면?**
  * 교착 상태가 아니다!
* **그래프에 사이클이 있다면** 시스템은 교착상태일 수도 있고 아닐 수도 있다.
  * 만약 한 자원에 인스턴스가 한 개씩 있을 경우에는 **교착 상태이다.**
  * 만약 한 자원에 인스턴스가 두 개 이상 있을 경우에는 **교착 상태가 될 가능성이 있다.**



## 교착 상태 처리 방법

* 시스템이 교착 상태가 되지 않도록 보장 **(미리 교착 상태가 발생할지 확인)**
  * **교착 상태 예방 (prevention) 기법** 
  * **교착 상태 회피 (avoidance) 기법**
* 시스템이 교착 상태가 될 수 있도록 허용하되 **회복시키는 방법**
  * 교착 상태 탐지 알고리즘 & 교착 상태로부터의 회복 알고리즘
* 교착 상태 문제를 무시하고 **교착 상태가 발생하지 않는다고 가정**
  * 성능 저하시 시스템을 중단하고 인위적으로 재시작한다.
  * **가장 많이 사용.**



## 교착 상태 처리 방법 - 교착 상태 예방

교착 상태가 발생하는 4가지 조건들 중 하나라도 성립하지 않도록 보장함으로써, 교착 상태의 발생을 예방.

* 배제, 점유와 대기, 비선점, 순환 대기



## 교착 상태 처리 방법 - 교착 상태 회피

교착 상태 예방은 장치의 속도와 처리율이 저하된다.

* 요청할 각 형태의 자원마다 최대 요청 정보를 미리 파악할 수 있다면 시스템이 교착상태가 되지 않는 알고리즘. **효율이 떨어진다.**



## 교착 상태 탐지 및 복수

* 교착 상태가 발생했는지 결정하기 위해 시스템의 상태를 검사하는 알고리즘
* 교착 상태로부터 복구하는 알고리즘

> 연산 시간이 부담이 크고 CPU 이용률이 저하된다.
