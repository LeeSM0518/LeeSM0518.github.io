# Basic Concepts (기본 개념들)

다중 프로그래밍은 **CPU 이용률을 최대화** 하기위해 항상 실행중인 프로세스를 가지게 해야한다.



## CPU - 입출력 버스트 사이클 (1)

프로세스의 실행은 CPU 실행과 입출력 대기의 사이클로 구성된다.

## CPU 스케줄러

CPU 가 Idle (유휴) 상태 일때, 운영체제는 **준비 완료 큐에 있는 프로세스들 중에서 하나를 선택해 실행** 해야 한다.

> 준비 완료 큐는 반드시 **선입 선출 방식의 큐는 아니다. (형태가 다양하다)**



## 선점 스케줄러 (1)

**스케줄링이 발생하는 시점들**

1. 실행 => 대기
2. 실행 => 준비완료
3. 대기 => 준비완료
4. 프로세스 종료

<img src="../capture/스크린샷 2019-06-11 오후 5.23.19.png" width=500>

> **비선점(nonpreemptive)** 스케줄링 : CPU가 뺏기지 않는다. 1, 4 에서만 발생
>
> **선점(preemptive)** 스케줄링 : CPU를 뺏긴다. 비선점 이외의 모든 스케줄링



## 선점 스케줄링 (2)

* **타이머** 와 같은 특정 하드웨어 요구
* **동기화** 를 위한 비용 증가



**선점은 운영체제 커널 설계에 영향을 준다.**

: OS가 실행 중에도 더 중요한 프로세스에게 CPU를 뺏길 수 있다.



## 디스패처

: CPU 스케줄러의 한 부분

* 문맥 교환
* 사용자 모드 전환
* 사용자 프로그램의 적절한 위치로 이동

> **디스패치 지연 (dispatch latency)** : 디스패치가 한 프로세스를 정지하고 다른 프로세스의 수행을 시작하는 데까지 소요되는 시간



# Scheduling Criteria (스케줄링 기준)

**CPU 스케줄링 알고리즘의 비교 기준들**

* **CPU 이용률 (utilization)**
* **처리량 (throughput)** : 단위 시간당 완료된 프로세스의 개수
* **총처리 시간 (turnaround time)** : 메모리에 들어가기 위해 대기한 시간 + 준비 완료 큐에서 대기한 시간 + CPU에서 실행한 시간 + 입출력 시간을 합한 시간
* **대기 시간 (waiting time)** : 준비완료 큐에서 대기하는 시간
* **응답 시간 (response time)** : 요구를 제출하고 첫 번째 응답이 나올 때까지의 시간

> **CPU 이용률과 처리량을 최대화하고 총처리 시간, 대기 시간, 응답 시간을 최소화하는 것이 바람직하다.**



# Scheduling Algorithms

* 선입 선처리 (First-come, First-served) 스케줄링
* 최단 작업 우선 (Shortest-job-first) 스케줄링
* 우선 순위 (Priority) 스케줄링
* 라운드-로빈 (Round-robin) 스케줄링
* 다단계 큐 (Multi-level Queue) 스케줄링
* 다단계 피드백 큐 (Multilevel Feedback Queue) 스케줄링



## 선입 선처리 스케줄링 (1)

**FCFS(First-Come, First-Served)** : 가장 간단한 CPU 스케줄링 알고리즘

* CPU를 먼저 요청하는 프로세스가 CPU를 먼저 할당받는다.

<img src="../capture/스크린샷 2019-06-11 오후 7.49.42.png">

* **비선점형** : CPU를 해제할 때까지 계속 CPU 점유
* **호위 효과 (convoy effect) 발생** : CPU를 양도하기를 기다리는 현상



## 최단 작업 우선 스케줄링 (1)

**SJF (Shortest-Job First)** : CPU가 이용 가능해지면, 다음 CPU 버스트가 **가장 짧은 프로세스에게 할당**

<img src="../capture/스크린샷 2019-06-11 오후 7.53.05.png">



## 최단 작업 우선 (2)

* **최소의 평균 대기 시간을 가지므로 최적**
* 최단 작업 우선 알고리즘은 **선점권이 있거나 없을 수 있다.**
  * 선점형 최단 작업 우선 알고리즘 = **최소 잔여시간 우선 스케줄링 (Shortest Remaining Time First, SRTF)**



## 최소 작업 우선 (3)

<img src="../capture/스크린샷 2019-06-11 오후 7.57.06.png">

> 비선점형과 선점형을 보면 문맥 시간으로 인해 선점형이 더 길며 늦게 끝난다는 것을 알 수 있다.



## 최소 작업 우선 (4)

**문제점**

* 다음 CPU 요청 시간을 파악하는 것은 불가능하다. **(버스트 시간을 실제로 미리 알 수 없다.)**
  * 다음 CPU 버스트가 앞의 버스트와 길이가 유사하다고 간주
  * 하지만, 구현X => CPU가 너무 많은 계산이 필요하여 속도가 저하된다.

## 우선 순위 스케줄링 (1)

우선 순위가 각 프로세스에게 주어지고 CPU는 **가장 높은 우선 순위를 가진** 프로세스에 할당된다.

## 우선 순위 스케줄링 (2)

* 선점형이거나 비선점형

  * 선점형 : 현재 실행되는 프로세스의 **우선 순위보다 높으면** CPU 선점
  * 비선점형

* **무한 봉쇄 (indefinite blocking) 문제의 발생 (또는 기아 상태 (starvation))**

  * 높은 우선 순위의 프로세스들이 **꾸준히 들어와서** 낮은 우선 순위의 프로세스들이 **CPU를 사용하지 못하게** 될 수도 있다.

  * **에이징(aging)**

    : 오래 대기하는 프로세스들의 우선 순위를 **점진적으로 증가시킨다.**

# Thread Scheduling

운영체제에서는 스케줄 되는 대상으 프로세스가 아니라 **커널 수준 스레드** 이다.



## 경쟁 범위

* **프로세스 경쟁 범위 (process-contention scope: PCS)**
  * 동일한 프로세스에 속한 스레드들 사이에서 CPU 경쟁
* **시스템 경쟁 범위 (system-contention scope: SCS)**
  * CPU 상에 어느 커널 스레드를 스케줄링 할 것 인지 결정



# Multiple Processor Scheduling

다중 처리기 스케줄링

* **대칭 다중 처리 (Symmetric multiprocessing: SMP)**
  * 각 처리기가 독자적으로 스케줄링



## 처리기 친화성 (1)

**처리기 친화성 (Processor affinity)** : 대부분의 SMP 시스템은 한 처리기에서 다른 처리기의 이주를 피하고 <u>같은 처리기에서 프로세스를 실행</u> 시키려 한다.



* **연선 친화성 (soft affinity)**
  * 프로세스가 처리기 사이에서 <u>이주하는 것이 가능하다</u>
* **강성 친화성 (hard affinity)**
  * 프로세스가 다른 처리기로 <u>이주하지 않게 지정 가능하다</u>



## 처리기 친화성 (2)

비균등 메모리 접근 (Non-Uniform Memory Access, NUMA)

: CPU가 다른 곳의 memory 에 **접근할 수 있거나 없을 수 있다.**

<img src="../capture/스크린샷 2019-06-12 오후 1.23.39.png" width=500>

> 프로세스를 같은 CPU에 할당할려고 노력



## 부하 균등화

SMP 시스템은 부하를 모든 처리기에게 균등하게 배분하는 것이 매우 중요하다.

=> **해야할 일을 동등하게 일 할 수 있도록 해야한다.**



* **부하 균등화 (load balancing)**
  * push 이주 : 부하가 불균형 상태이면 과부하 처리기에서 쉬고 있거나 덜 바쁜 처리기로 프로세스 이동
  * pull 이주 : 쉬고 있는 처리기가 바쁜 처리기를 기다리고 있는 프로세스를 pull



## 다중 코어 프로세서 (1)

다중 코어 프로세서는 스케줄링 문제를 **복잡하게 한다.**

* **메모리 멈춤 (memory stall)** 
  * 프로세스가 **메모리를 접근할 때** 데이터를 기다리면서 많은 시간을 허비

<img src="../capture/스크린샷 2019-06-12 오후 1.32.39.png" width=500>



* 다중 스레드 프로세서 코어의 구현 => 한 스레드가 **메모리를 기다리면서 멈추면 , 코어는 다른 스레드로 전환한다.**

# Real-Time CPU Scheduling

실시간 CPU 스케줄링



## 실시간 컴퓨팅의 두 가지 타입

* **연성 실시간 시스템 (Soft Real-time System)**
  * 덜 엄격한 시간 제약조건
  * 실시간 테스크가 높은 우선순위를 부여 받아 완결될 때까지 그 우선순위를 유지하기만 하면 된다.
* **경성 실시간 시스템 (Hard Real-time System)**
  * 마감시간 안에 반드시 완수해야 하는 가장 엄격한 제약 조건
  * 중요한 task를 정해진 시간 내에 처리 완료



## 지연 시간 최소화

**사건 지연시간 (Event latency)**

* 사건이 발생해서 그에 맞는 서비스가 실행될 때까지의 시간

  <img src="../capture/스크린샷 2019-06-12 오후 1.45.07.png">

  * 인터럽트 지연 (interrupt latency) : 이벤트를 실행 준비하는 시간
  * 디스패치 지연 (dispatch latency) : 이벤트 처리 시간



## 프로세스의 특성

주기 프로세스 (periodic task)

* 일정한 간격으로 CPU를 필요
* p : 주기
* d : 마감 시간
* t : 처리시간 = 요청시간

## Rate-monotonic 스케줄링

선점 가능한 정적 **우선 순위** 정책을 이용하여 주기 태스크들을 스케줄

<img src="../capture/스크린샷 2019-06-12 오후 1.53.59.png">

> 50 주기 안에 성공적으로 P1을 실행하지 못한다.



* **Rate-monotonic scheduling**
  * 주기에 따라, 우선순위 결정 => 주기가 짧은 태스크는 높은 우선순위, 주기가 길면 낮은 우선순위
  * P1이 P2 보다 높은 우선 순위를 가진다.

<img src="../capture/스크린샷 2019-06-12 오후 1.56.21.png">



<img src="../capture/스크린샷 2019-06-12 오후 1.57.27.png">



* **Earliest deadline first scheduling**

  * 마감시간에 따라서 우선순위를 동적으로 부여한다.

    => 마감시간이 빠를수록 우선순위는 높아지고, 늦을수록 낮아진다.

  * 마감시간이 주기와 가까울수록 우선순위가 높다
